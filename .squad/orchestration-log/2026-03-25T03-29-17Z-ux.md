# UX — Dashboard Mobile Audit

**Timestamp:** 2026-03-25T03-29-17Z
**Mode:** Read-only system audit
**Scope:** Dashboard views, styles, mobile responsive design, shell/navigation patterns, HTMX fragment behavior, existing test coverage

## Audit Summary

Examined dashboard mobile system across:
- Dashboard layout shell (`src/dashboard/views/layout.ts`, `src/dashboard/public/styles.css`)
- Core operator pages (`article.ts`, `publish.ts`, `runs.ts`, `memory.ts`, `config.ts`, `agents.ts`, `new-idea.ts`)
- Shared patterns (headers, action groups, data tables, navigation, toolbars)
- Test coverage for mobile hooks and responsive behavior

## Key Findings

1. **Shell-level failures** — Primary nav (`.header-nav`), sticky header, and page layout collapse inconsistently across breakpoints
2. **Repeated patterns** — Same data-table, action-group, and filter patterns used across multiple pages with no centralized mobile contract
3. **HTMX fragment mismatch** — Swapped content from partial renders doesn't inherit shell mobile behavior; needs wrapper scoping
4. **Mobile-unsafe selectors** — `.agent-grid` overloaded; `.card-actions`, `.quick-actions`, `.preview-toolbar` lack stacking/wrapping rules
5. **Test gap** — Current dashboard tests focus on route/copy validation; no viewport-specific assertions for mobile hooks or fragment structure

## Decision Inbox Entry

Authored decision: "Dashboard Mobile Fixes Must Start at Shared Shell and Fragments" → inbox merge pending.

## Handoff Notes

- Mobile refactoring should be **system-wide** (shell + shared primitives), not page-by-page
- HTMX fragment-level classes critical for consistency across partial updates
- New tests must assert mobile behavior at the fragment/component level, not only full-page level
