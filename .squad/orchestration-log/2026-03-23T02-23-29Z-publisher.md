## Key Findings

1. **Draft Creation**: Endpoint creates draft via Substack POST, returns draft URL
2. **Draft Updates**: Subsequent calls with same `draft_url` update existing draft instead of creating new
3. **Pipeline State**: `pipeline.db` tracks draft URLs for re-editing workflows
4. **Publish-Now**: Minimal surface; standard flow always goes through draft

---

## Architecture Notes

- Draft URL stored in database enables edit/republish workflows
- Create vs. update logic gated by presence of prior draft URL
- Team can update drafts without creating duplicates
