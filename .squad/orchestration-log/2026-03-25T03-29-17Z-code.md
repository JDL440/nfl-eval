# Code — Dashboard Mobile Audit

**Timestamp:** 2026-03-25T03-29-17Z
**Mode:** Read-only system audit
**Scope:** Dashboard server/HTMX architecture, CSS/responsive system, fragment-level structure, shared utilities, test infrastructure

## Audit Summary

Examined dashboard mobile architecture across:
- Server-side rendering seams (`src/dashboard/views/` route handlers, HTMX partial responses)
- Responsive CSS system (`src/dashboard/public/styles.css`, breakpoint hooks)
- Fragment-level structure (header/nav classes, action-group markup, data-table wrappers)
- Shared utilities for DOM/layout helpers
- Dashboard test suite (route coverage, HTMX assertions, mobile/responsive coverage gaps)

## Key Findings

1. **Server-side render contract unclear** — Dashboard views emit fragments without explicit mobile-aware wrappers; `.header-nav`, `.page-header`, `.card-actions` markup assumes desktop presentation
2. **Shared CSS primitives missing** — Breakpoints exist but key dashboard elements (tables, action groups, filters) lack stacking/wrapping rules for <640px
3. **Fragment inheritance gap** — HTMX swaps (`article.ts` detail panels, `publish.ts` toolbar, `runs.ts` table rows) inherit parent shell mobile behavior unpredictably
4. **Class scoping conflicts** — `.agent-grid` used in two different contexts (`agents.ts` selector grid vs `new-idea.ts` chip display) with no mobile override strategy
5. **Test infrastructure** — Dashboard tests mock DOM/routes but don't assert breakpoint-specific rendering or verify mobile fragment-level classes

## Technical Recommendations

- Extract shared mobile wrappers for dashboard fragments (`artifact-table`, `action-group`, `memory-table`)
- Add `.mobile-*` and `.touch-*` modifier classes to enable context-specific mobile presentation
- Implement fragment-level assertions in test suite (`testMobileFragment()` helper)
- Scope `.agent-grid` overloads with page context (data attributes or parent class)
