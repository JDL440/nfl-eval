## Key Findings

### 1. **Ambiguous "Publish Workspace" Term**
- Used exactly once (article.ts:513) in disabled-state tooltip
- Mental model mismatch — unclear if it refers to `/articles/:id/publish` page or article detail stage
- No other UI context reinforces the term

### 2. **Two-Step Workflow Clarity**
- UI must clearly surface Draft → Publish sequence
- Current copy: *"Create a Substack draft in the publish workspace before publishing"*
- Better: *"Create a draft on Substack, then publish it live from here"*

### 3. **Button Label Ambiguity**
- "Publish to Substack" doesn't clarify whether it creates a new draft or publishes existing
- Consider: "Publish Draft to Substack" or "Go Live" once draft exists

### 4. **Weak Copy in Success States**
- Draft creation succeeds, but button label remains static
- User needs explicit confirmation that next action (Publish) is now available

### 5. **"Publish All" Disabled State Redundancy**
- Button is only shown when draft exists, yet tooltip still says "Create a draft first"
- Unnecessary cognitive load

## Recommendations (Summary)

| Action | File | Current | Proposed |
|--------|------|---------|----------|
| Rename term | article.ts:532 | "Open Publish Workspace" | "Open Publish Preview" |
| Clarify workflow | article.ts:511 | "Create a draft in publish workspace" | "Create Substack draft, then publish" |
| Update button labels | publish.ts | "Publish to Substack" | "Publish Draft to Substack" |
| Remove redundancy | publish.ts:359 | Disabled state w/ tooltip | Remove when `publishEnabled` true |
