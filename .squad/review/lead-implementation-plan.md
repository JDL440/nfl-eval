### 1b. Tool Wiring Detail

| Surface | How tools are registered | How tools are invoked |
|---|---|---|
| **mcp/server.mjs** (local tools) | Imports `registerLocalTools()` from `tool-registry.mjs`. Each entry has `{name, inputSchema (zod/v4), handler}`. Schema is passed as raw Zod shape to `server.registerTool()`. | Copilot CLI calls via stdio MCP transport. Handler receives parsed args, returns `normalizeToolResult()` → `{content, isError}`. |
| **src/mcp/server.ts** (pipeline) | Hand-defined `Tool[]` with JSON Schema `inputSchema`. Registered via `CallToolRequestSchema` + switch-case on tool name. | Copilot CLI or `tsx src/cli.ts mcp-pipeline`. Switch-case dispatches to inline logic calling `repo`, `engine`, or `autoAdvanceArticle()`. |
| **actions.ts → runner.ts → gateway.ts** | No tool registration — this is the agent execution path. `StageActions` call `ctx.runner.run()` which calls `this._gateway.chat()`. | `executeTransition()` → `STAGE_ACTIONS[name]()` → `runner.run()` → `gateway.chat()`. Provider/model resolved via `ModelPolicy` + `AGENT_STAGE_KEY` map. |

### 1c. Provider Allowlist Flow

```
LLM_PROVIDERS env → resolveStartupProviderIds() → registers providers
                                                            ↓
Article.preferred_llm_provider → buildArticleProviderHint() → preferredProvider + 'prefer'
                                                            ↓
AgentRunParams.allowedProviders → ChatRequest.allowedProviders
                                                            ↓
gateway.buildAttempts() filters availableProviders by allowedProviders set
```

**Key finding:** `allowedProviders` is plumbed from `AgentRunParams` → `ChatRequest` → `buildAttempts()`, but **no stage action ever populates `allowedProviders`**. Only `preferredProvider` + `providerStrategy: 'prefer'` is used via `buildArticleProviderHint()`. The `allowedProviders` field exists for external callers (dashboard API) but is never exercised by the automated pipeline.

## 2. Identified Gaps and Risks

### 2a. Architectural Gaps

| # | Gap | Severity | Location |
|---|-----|----------|----------|
| G1 | **Two disconnected MCP servers** — `mcp/server.mjs` (data/media tools) and `src/mcp/server.ts` (pipeline ops) are separate processes with no shared state. A Copilot CLI session sees both but they cannot call each other. | Low | Architecture |
| G2 | **Pipeline MCP server uses two SDK versions** — `mcp/server.mjs` uses `McpServer` (high-level SDK), `src/mcp/server.ts` uses `Server` (low-level SDK). Both from `@modelcontextprotocol/sdk` but different API surfaces. | Low | Consistency |
| G3 | **tool-registry.mjs uses `zod/v4` path** — imported as `import * as z from "zod/v4"` while core TypeScript uses standard `zod` imports. This works because the package bundles both, but is a fragility point if Zod is updated. | Medium | `mcp/tool-registry.mjs:1` |
| G4 | **No tool-call capability in agent runs** — `AgentRunner.run()` sends messages to LLM but provides no tool definitions. Agents cannot call MCP tools mid-run; all context is pre-gathered via `gatherContext()`. This is by design but limits agents to single-turn text generation. | Informational | `runner.ts:396-415` |
| G5 | **`allowedProviders` on `AgentRunParams` is dead code in pipeline** — Field exists and is wired through to gateway, but no `StageAction` ever sets it. Only external callers could use it. | Low | `runner.ts:59`, `actions.ts` |
| G6 | **No input validation on pipeline MCP tools** — `src/mcp/server.ts` casts `args` directly without schema validation (`args.article_id as string`). Local-tools server validates via Zod schemas. | Medium | `src/mcp/server.ts:260-262` |

### 2b. Operational Risks

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| R1 | **Silent skill exclusion** — `runner.run()` silently drops missing skills (line 366). If a skill file is renamed or deleted, the agent runs with degraded context and no warning. | Medium | Add warn log when skill not found |
| R2 | **Unbounded memory recall** — `this.memory.recall(agentName, { limit: 10 })` is hardcoded. For prolific agents this could inject stale/irrelevant context. | Low | Already bounded at 10; acceptable |
| R3 | **gatherContext() throws on missing primary artifact** — `readArtifact()` throws if primary is missing, but there's no guard before calling it. A corrupted DB could crash the pipeline. | Medium | `actions.ts:731` |
| R4 | **`autoAdvanceArticle` revision loop** — If both editor and writer loop deterministically, `maxRevisions` (default 3) prevents infinite loops, but the `maybeEscalateRepeatedRevisionBlocker()` logic adds a second safety net. This is well-designed. | Low | Already mitigated |
| R5 | **LM Studio hijack in auto mode** — `FakeLmStudioProvider.supportsModel()` returns `true` for everything. Gateway handles this with `canAutoRouteModel()` check, but only when `hasNonLmStudioProvider` is true. If LM Studio is the sole provider, it routes everything there. | Low | By design for local-only setups |

## 5. Breaking Changes Assessment

**No breaking changes identified.** All proposed changes are additive:

- Adding tests does not change behavior
- Adding console.warn for missing skills is a log-level change only
- Adding input validation to MCP server would reject previously-silently-wrong inputs, but these would have failed downstream anyway
- Zod import normalization requires verification but is functionally equivalent

### Backward Compatibility Notes

- `AgentRunParams.allowedProviders` remains optional and unused by pipeline actions — safe
- `preferredProvider` + `providerStrategy: 'prefer'` is the active provider routing path — unchanged
- `AGENT_STAGE_KEY` mapping determines model tiers — unchanged
- Two-server MCP architecture is stable and intentional — no consolidation needed

---

## Summary

The three-component architecture (MCP → Actions → Runner → Gateway) is **correctly wired** with clean separation of concerns. The primary technical debt is in test coverage (especially `allowedProviders` filtering and pipeline MCP server) and minor defensive coding gaps (silent skill drops, unvalidated MCP args). No emergency fixes are required. The recommended P1-P4 improvements can be done incrementally without risking the existing pipeline.
